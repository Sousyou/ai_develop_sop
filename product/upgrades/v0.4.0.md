# Upgrade To 0.4.0

## Target Version
`0.4.0`

## Purpose

Use this upgrade when a copied project needs to align its SOP assets with the current Dev SOP package.
This note is written so an AI agent can execute the upgrade directly inside a real project repository.

## Applies To

- copied projects on `0.3.x`
- copied projects that still use `ai/` as the SOP namespace root
- copied projects that still use a root-level `project/` control surface from older starter structure

## Summary

- rename the SOP namespace root from `ai/` to `.dev_sop/`
- move the recovery/control surface to `.dev_sop/control/`
- narrow `.dev_sop/control/` so it contains only `CURRENT.md` and `DOC_MAP.md`
- keep only starter-owned `example-*` reference specs plus project-owned real specs in `.dev_sop/doc/specs/`
- add explicit SOP versioning and upgrade notes via `.dev_sop/VERSION.md` and `.dev_sop/upgrades/*`

## End-State Contract

After the upgrade:

- the SOP namespace root is `.dev_sop/`
- `.dev_sop/control/` contains only `CURRENT.md` and `DOC_MAP.md`
- `.dev_sop/doc/specs/` contains:
  - `README.md`
  - starter-owned `example-*` reference specs
  - project-owned real task specs
- there is no starter-owned change-history spec left in `.dev_sop/doc/specs/`
- `.dev_sop/VERSION.md` exists and states `0.4.0`
- `.dev_sop/upgrades/` exists and contains versioned upgrade notes

## Preflight

Before editing:

1. Read the project's `.dev_sop/VERSION.md` if it exists.
2. Inspect `.dev_sop/doc/specs/` and separate project-owned real specs from starter-owned examples or old starter change-history specs.
3. Inspect whether the project still uses `ai/` or a root-level `project/` control surface.
4. Identify any project-local SOP customizations in guides, skills, facts, or templates that should be preserved.

## Required Changes

1. Rename the SOP namespace root from `ai/` to `.dev_sop/`, and update all live path references in README, AGENTS, adapters, guides, templates, skills, and facts.
2. Replace any root-level `project/` control surface from older starter copies with `.dev_sop/control/CURRENT.md` and `.dev_sop/control/DOC_MAP.md`.
3. Remove `.dev_sop/control/README.md`, `.dev_sop/control/decisions/`, and `.dev_sop/control/experiments/` if they exist from earlier starter versions.
4. In `.dev_sop/doc/specs/`, preserve starter-owned `example-*` reference specs and preserve project-owned real task specs.
5. Delete only starter-owned change-history specs that describe the SOP starter's own evolution rather than reusable spec examples.
6. Add `.dev_sop/VERSION.md` and `.dev_sop/upgrades/*`.
7. Update docs so durable reusable decision rationale and experiment outcomes route to facts, while task-local detail stays in specs or change summaries.

## Preserve Rules

Do preserve:

- project-owned real task specs under `.dev_sop/doc/specs/`
- starter-owned `example-*` reference specs used as writing examples
- project-local SOP customizations that do not conflict with the target version's structure

Do not preserve:

- old root-level `project/` control-surface artifacts copied from older starter versions
- starter-owned change-history specs that describe how the starter repository itself evolved
- obsolete control ledgers such as `.dev_sop/control/decisions/` or `.dev_sop/control/experiments/`

## Execution Notes For AI

- Treat `.dev_sop/VERSION.md` as the version source of truth.
- Do not delete a spec just because it has a date-based filename; decide whether it is:
  - a starter-owned reference example
  - a starter-owned change-history artifact
  - a project-owned real task spec
- When uncertain whether a spec is project-owned, preserve it and surface the ambiguity.
- Keep project-specific SOP customizations unless they directly conflict with the target structure.
- Update path references and wording in the project's live SOP docs after the structural move.

## Validation

- no live SOP docs still point to `ai/` or the old root-level `project/` control surface
- `.dev_sop/control/` contains only `CURRENT.md` and `DOC_MAP.md`
- `.dev_sop/doc/specs/` contains only `README.md`, starter-owned `example-*` reference specs, and project-owned real task specs
- `.dev_sop/VERSION.md` exists and states `0.4.0`

## Completion Rule

The upgrade is complete only when:

1. structure matches the end-state contract
2. live SOP docs reference the new paths and routing rules
3. preserved project-owned specs and customizations are still intact
4. the project's `.dev_sop/VERSION.md` is updated to `0.4.0`

## Merge Cautions

- do not delete project-owned specs just because they live under `.dev_sop/doc/specs/`
- do not delete starter-owned `example-*` specs that are still intended as writing references
- do not overwrite project-local SOP guidance or skills unless the project explicitly wants to rebase onto the starter's newer rule set
- do not treat the upgrade notes as a substitute for reviewing local customizations
