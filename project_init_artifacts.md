# 新项目初始化标准产物清单

- **版本**：v1.0
- **状态**：第一版
- **定位**：定义新项目在完成初始化时，最少应生成哪些目录与文件，作为 `project_bootstrap_sop.md` 的标准产物清单。
- **适用范围**：后续所有按当前 AI 项目 SOP 初始化的新项目。

---

## 1. 文档目标

本清单用于回答以下问题：

1. 新项目初始化完成时，最少应生成哪些治理类文件和目录。
2. 哪些产物是固定必选项，哪些产物是按需启用项。
3. 如何判断初始化产物是否已经达到“可进入开发”的标准。

本文件只定义初始化产物，不定义任务执行、事实回写和维护纠偏流程本身。

---

## 2. 产物分层

新项目初始化的最小产物，分为四层：

1. 项目入口层
2. 项目规则层
3. 项目事实层
4. 技能沉淀层

如项目需要承载实际代码开发，再按项目形态补充最小业务目录骨架。

---

## 3. 固定必选产物

以下产物为所有新项目初始化时默认应生成的固定项。

### 3.1 项目入口层

- `README.md`
- `AGENTS.md`

### 3.2 项目规则层

- `rules/`
- `rules/workflow_rules.md`
- `rules/boundaries_rules.md`
- `rules/validation_rules.md`
- `rules/scope_control_rules.md`
- `rules/writeback_rules.md`

### 3.3 项目事实层

- `docs/`
- `docs/facts/`
- `docs/facts/project_scope.md`
- `docs/facts/facts_index.md`
- `docs/facts/golden_cases.md`
- `docs/facts/task_template.md`

### 3.4 技能沉淀层

- `skill_policy.md`
- `skill_registry.md`
- `skill_template.md`

---

## 4. 按需启用产物

以下产物不是所有项目初始化时都必须生成，而是根据项目形态按需补充。

### 4.1 代码目录骨架

只有当项目在初始化后将立即进入代码开发阶段时，才创建最小代码目录。

默认要求：

- 仅创建当前 phase 明确需要的目录
- 目录结构选择应参考目录规范
- 不为未来能力预铺完整分层

常见按需目录示例：

- `app/`
- `src/`
- `tests/`
- `scripts/`

### 4.2 运行入口

只有当项目已明确需要对外入口时，才创建对应入口文件或目录。

常见按需入口示例：

- API 入口
- CLI 入口
- Worker 入口

### 4.3 部署与环境文件

只有当当前 phase 已明确需要运行或交付环境时，才创建。

常见按需示例：

- `.env.example`
- `pyproject.toml`
- `requirements.txt`
- 容器或部署目录

---

## 5. 最小业务目录骨架规则

若项目需要立刻进入代码开发，业务目录骨架应遵守以下规则：

1. 先根据项目形态判断是否真的需要代码目录。
2. 只创建支撑当前 phase 的最小目录。
3. 目录结构应服从目录规范，而不是凭感觉铺满。
4. 若尚未明确分层，不应提前创建复杂分层目录。

换句话说，初始化阶段固定生成的是治理骨架；业务骨架则按需最小生成。

---

## 6. 初始化完成判定

当以下条件全部满足时，可认为标准产物清单已落地：

1. `README.md` 已建立。
2. `AGENTS.md` 已建立，且具备最小必要内容。
3. `rules/` 已建立，且五份规则文档齐备。
4. `docs/facts/` 已建立，且四份 facts 模板齐备。
5. 三份 skill 文档已建立。
6. 若项目当前 phase 需要代码开发，最小业务目录骨架也已建立。

---

## 7. 与其他文档的关系

- 初始化流程本身由 `project_bootstrap_sop.md` 定义。
- 总体文档关系由 `ai_project_sop.md` 定义。
- 代码目录如何按需生成，应进一步参考目录规范。

---

## 8. 最终结论

新项目初始化的最小标准，不是先把业务代码目录铺满，而是先把治理骨架补齐：

> **入口文件、规则层、facts 层、skill 层先落地，业务目录按需最小生成。**

只有这样，后续项目创建才能真正被 SOP 稳定驱动。
