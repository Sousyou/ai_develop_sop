# Upgrade To 1.1.0

## Target Version
`1.1.0`

## Purpose

Use this upgrade when a copied project already adopted `1.0.0` but still has duplicate-function surfaces:

- `.dev_sop/core/rules/*`
- `.dev_sop/tools/*`
- `.dev_sop/skills/*`

This upgrade simplifies the package to:

- `.dev_sop/core/*` for shared reusable assets
- `.dev_sop/rules/*` for the single active rule surface
- `.dev_sop/control/*`, `.dev_sop/specs/*`, `.dev_sop/facts/*` for execution assets
- root `AGENTS.md` and `CLAUDE.md` as the only tool-entry adapters

## Applies To

- copied projects on `1.0.0`
- copied projects with the `1.0.0` split model that still keep the duplicate-function trees above

## End-State Contract

After the upgrade:

- `.dev_sop/core/*` holds shared guides, templates, and reusable shipped skills
- `.dev_sop/rules/*` is the only active rule surface
- `.dev_sop/control/*` holds `CURRENT.md` and `DOC_MAP.md`
- `.dev_sop/specs/*` holds task specs
- `.dev_sop/facts/*` holds project facts
- `AGENTS.md` and `CLAUDE.md` are the only mainline entry adapters
- there is no active `.dev_sop/tools/*`, `.dev_sop/core/rules/*`, or `.dev_sop/skills/*` tree
- the precedence contract is `approved exceptions > active task spec > repository rules > entry adapter notes`
- `.dev_sop/VERSION.md` exists and states `1.1.0`

## Required Moves

Apply these path migrations:

- `.dev_sop/core/rules/* -> merge into .dev_sop/rules/*`
- `.dev_sop/tools/* -> remove; move any still-needed entry notes into AGENTS.md or CLAUDE.md`
- `.dev_sop/skills/* -> remove; move any still-needed local operating guidance into .dev_sop/rules/* or, only when it is truly reusable package behavior, into .dev_sop/core/skills/*`

## Required Changes

1. Rewrite `AGENTS.md` and `CLAUDE.md` so they are the only tool-entry adapters.
2. Rewrite `.dev_sop/README.md` and `.dev_sop/VERSION.md` to describe the simplified model.
3. Make `.dev_sop/rules/rule-index.md` the authoritative entry for both default and project-local rules.
4. Update templates, guides, control docs, and facts so they point to `.dev_sop/rules/*` and `.dev_sop/core/skills/*`.
5. Remove the duplicate-function trees from the mainline package.
6. Update the validation rules so they explicitly check that the removed trees do not reappear.
7. Update `.dev_sop/VERSION.md` to `1.1.0` only after the migration validates.

## Preserve Rules

Do preserve:

- real task specs under `.dev_sop/specs/*`
- stable facts under `.dev_sop/facts/*`
- local operating rules that still make sense under `.dev_sop/rules/*`
- reusable shipped skills that still belong under `.dev_sop/core/skills/*`

Do not preserve:

- active `.dev_sop/tools/*`, `.dev_sop/core/rules/*`, or `.dev_sop/skills/*` trees
- duplicated rule text split across adapters, rule files, and removed trees
- local workflow fragments that should now be expressed as rule updates instead of a second skill surface

## Validation

- `.dev_sop/core/*` remains the shared reusable asset root
- `.dev_sop/rules/*` is the single active rule surface
- `.dev_sop/control/*`, `.dev_sop/specs/*`, and `.dev_sop/facts/*` remain the execution asset roots
- `.dev_sop/core/skills/*` remains the only active skill surface
- `AGENTS.md` and `CLAUDE.md` are the only mainline entry adapters
- `.dev_sop/rules/rule-index.md` exists
- `.dev_sop/core/templates/task-spec-template.md` still includes:
  - `Applicable Project Rules`
  - `Authoritative References`
  - `Approved Exceptions`
  - `Project Validation Gates`
- no current mainline docs require `.dev_sop/tools/*`, `.dev_sop/core/rules/*`, or `.dev_sop/skills/*`
- `.dev_sop/VERSION.md` exists and states `1.1.0`

## Completion Rule

The upgrade is complete only when:

1. the copied project matches the simplified end-state contract
2. preserved project-local customizations are still intact
3. `.dev_sop/VERSION.md` is updated to `1.1.0`
