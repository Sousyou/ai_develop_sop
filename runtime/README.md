# Runtime Directory

- **Version**: `v2.0`
- **Status**: official
- **Role**: documents the runtime instance files generated for injected host projects

---

## Purpose

`runtime/` belongs to the portable SOP engine, but the runtime instance files themselves belong to injected host projects.

It answers:

1. which runtime instance files are generated by init
2. which runtime file is authoritative for each kind of instance state
3. how `session_brief` stays a short derived digest

---

## Source Vs Host Boundary

In the SOP engine source repository:

- `runtime/templates/*` is source content
- `runtime/*.md` instance files are not authoritative source-repository state
- if runtime instance files are generated here by mistake, delete them instead of treating them as source truth

In an injected host project:

- `runtime/entry_state.md`
- `runtime/project_mount.md`
- `runtime/project_charter.md`
- `runtime/current_target.md`
- `runtime/session_brief.md`

are generated runtime assets and become the authoritative host-project instance state.

---

## Generated Runtime Files

`ai_sop_init.bat` generates or resets:

- `runtime/entry_state.md`
- `runtime/project_mount.md`
- `runtime/project_charter.md`
- `runtime/current_target.md`
- `runtime/session_brief.md`

If the host project has no `README.md`, init also creates a thin bootstrap README at the host-project root.
`ai_sop_init.bat` should be run only after `ai_develop_sop/` has been placed under a real host-project directory.

---

## Runtime Rules

1. `runtime/entry_state.md` is the authoritative instance-state entry
2. `runtime/project_mount.md` is the authoritative mount-boundary entry
3. `runtime/project_charter.md` stores long-term direction
4. `runtime/current_target.md` stores high-frequency current goals
5. `runtime/session_brief.md` is a derived digest and must be co-updated in the same turn when its source runtime documents change
