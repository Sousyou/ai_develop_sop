# 项目初始化 SOP

- **版本**：v1.0
- **状态**：第一版
- **定位**：本文件用于定义新项目启动时的标准初始化流程，确保项目在开始开发前具备清晰边界、最小骨架和基础文档。
- **适用范围**：后续所有使用 AI 参与创建和启动的工程项目。

---

## 1. 文档目标

本 SOP 用于回答以下问题：

1. 新项目开始时，AI 应按什么顺序推进初始化。
2. 初始化阶段必须明确哪些边界信息。
3. 初始化阶段应创建哪些最小目录和文档。
4. 什么情况下可以认为项目已经完成启动，进入后续开发阶段。

本文件只负责项目初始化，不负责单任务执行、事实回写细则和维护恢复流程。

---

## 2. 初始化原则

### 2.1 先明确边界，再初始化结构

项目启动时，优先明确目标、phase、非目标和交付边界，再创建目录和文档。

### 2.2 最小骨架优先

初始化只创建当前阶段明确需要的最小目录与文档，不为未来预埋大而全结构。

### 2.3 初始化即建立治理入口

项目初始化不仅是创建目录，更要同时建立：
- 项目规则入口
- 项目事实入口
- 后续任务执行入口

### 2.4 项目初始化不等于开始实现功能

初始化阶段的目标是把项目启动条件建立起来，而不是提前开始功能开发、重构或平台化设计。

---

## 3. 初始化触发条件

以下场景进入项目初始化流程：

1. 全新项目启动。
2. 现有项目进入新的独立 phase，且需要重新明确边界和最小骨架。
3. 现有项目发生大范围重构，已经接近重新开项目。

以下场景通常不进入项目初始化流程：

1. 普通单任务开发。
2. 小范围 bug 修复。
3. 仅修改单个模块实现。

---

## 4. 初始化标准流程

项目初始化按以下顺序执行：

### 4.1 明确项目目标

必须先明确：
- 项目要解决什么问题
- 当前阶段的最小目标是什么
- 当前希望交付什么结果

要求：
- 目标必须可描述、可理解、可评审
- 避免把长期愿景直接当成当前 phase 目标

### 4.2 明确当前 phase

必须明确当前处于哪个阶段，例如：
- 原型验证
- MVP 构建
- 第一版可用工程
- 重构阶段

要求：
- phase 必须有边界
- 不允许同时承载多个不相干的大目标

### 4.3 明确非目标

初始化阶段必须明确当前不做什么。

要求：
- 非目标应写清楚，避免后续 scope 漂移
- 如果某能力不在当前 phase 内，应明确标记为 out-of-scope

### 4.4 判断项目形态

应判断当前项目属于哪类形态，以便选择最小骨架。

至少要识别：
- 是否为全新项目
- 是否需要工程化目录
- 是否需要 API / CLI / Worker 等入口
- 是否只是最小验证型项目

本步骤的目的不是分类越细越好，而是确保后续骨架选择有依据。

### 4.5 选择最小骨架

根据当前目标和项目形态，选择能支撑当前 phase 的最小目录结构。

要求：
- 只创建当前需求真正需要的目录
- 不为未来需求提前创建完整分层
- 目录结构必须足以承接后续开发任务

### 4.6 初始化基础文档

项目初始化阶段至少应创建以下文档入口：
- `README.md`
- `AGENTS.md`
- `docs/facts/project_scope.md`
- `docs/facts/facts_index.md`
- `docs/facts/golden_cases.md`
- `docs/facts/task_template.md`
- `rules/`
- `skill_policy.md`
- `skill_registry.md`
- `skill_template.md`

要求：
- 每份文档都必须有明确作用
- 只建立最小必要文档，不铺设过多模板
- skill 文档允许先创建最小骨架，但不得在初始化阶段预填大量未验证 skill
- 具体标准产物清单以 `project_init_artifacts.md` 为准

### 4.7 明确 `AGENTS.md` 的必要内容

在创建新项目或更新已有项目时，必须同时确认 `AGENTS.md` 是否已经具备项目运行所需的最小内容。

`AGENTS.md` 至少应明确以下内容：
- 当前项目或当前 phase 的目标
- 当前明确不做什么
- 当前项目边界与改动边界
- 当前默认执行风格与约束重点
- 当前验证预期
- 当前 facts 文档入口或事实沉淀要求
- 当前是否需要在任务结束后检查 SOP 更新

要求：
- 若是新项目，则初始化时直接建立最小可用版 `AGENTS.md`
- 若是已有项目，则初始化或 phase 更新时检查并补齐缺失内容
- `AGENTS.md` 只承载项目级长期有效规则，不承载单任务细节

### 4.8 `AGENTS.md` 最小骨架

项目初始化时，`AGENTS.md` 至少应具备以下骨架结构：

1. 文档定位
2. 当前项目目标
3. 当前明确不做什么
4. 项目边界与改动边界
5. 默认执行风格与约束重点
6. 当前验证预期
7. facts 入口与事实沉淀要求
8. SOP 更新检查规则
9. 文档使用顺序

以上内容的目的不是追求完整模板，而是确保 AI 在进入项目后，有最小可用的项目级规则入口。

要求：
- 本节定义的是初始化最低标准，不是完整模板大全
- 若当前 phase 不需要更细化内容，不应继续扩写 `AGENTS.md`
- 后续只有在项目边界、执行约束或事实入口发生稳定变化时，才补充 `AGENTS.md`

### 4.9 `AGENTS.md` 各部分的最小要求

#### 文档定位

至少要说明：
- `AGENTS.md` 是项目级规则入口
- 它承载的是长期有效、跨任务共享的规则
- 它不承载单任务细节

#### 当前项目目标

至少要说明：
- 当前项目总体目标
- 当前 phase 目标

#### 当前明确不做什么

至少要说明：
- 当前 phase 的非目标
- 当前明确排除的范围

#### 项目边界与改动边界

至少要说明：
- 当前项目主要处理什么
- 当前阶段允许修改哪些内容
- 当前阶段不应随意扩展到哪些内容

#### 默认执行风格与约束重点

至少要说明：
- 默认执行风格
- 当前最重要的执行约束

例如：
- 最小实现优先
- 不扩 scope
- 不 opportunistic refactor
- 不为未来做抽象

#### 当前验证预期

至少要说明：
- 当前项目如何判断一次任务的交付结果是合格的
- 当前阶段最关注哪些验证点

#### facts 入口与事实沉淀要求

至少要说明：
- facts 文档的入口位置
- 稳定事实应如何沉淀
- 事实与规则不能混写

#### SOP 更新检查规则

至少要说明：
- 任务结束后是否要检查 SOP 是否需要更新
- 什么情况下应更新 SOP 或 `AGENTS.md`

#### 文档使用顺序

至少要说明：
- AI 进入项目后先读什么
- 后续按什么顺序进入各类 SOP

### 4.10 明确后续开发入口

在初始化结束前，应明确后续开发将如何进入任务执行流程。

至少需要明确：
- 后续任务将遵循 `task_execution_sop.md`
- 稳定事实将按 `fact_writeback_sop.md` 沉淀
- 项目偏移时进入 `maintenance_recovery_sop.md`
- AI 执行行为将受 `rules/*` 约束
- 稳定重复流程将按 skill 文档体系沉淀

---

## 5. 初始化阶段的最小产物

初始化完成后，项目至少应具备以下产物：

### 5.1 项目边界

已明确：
- 当前项目目标
- 当前 phase
- 当前非目标
- 当前交付边界

### 5.2 最小目录骨架

已建立足以支撑当前 phase 的最小目录结构。

### 5.3 项目级规则入口

已建立或更新 `AGENTS.md` 作为项目级规则入口，且其最小必要内容已经明确。

### 5.4 项目事实入口

已建立 `docs/facts/` 作为项目事实沉淀入口，且最小模板已经具备。

### 5.5 后续执行路径

已明确后续开发将进入统一的任务执行 SOP，而不是直接无边界扩展。

### 5.6 执行护栏入口

已建立 `rules/` 作为 AI 执行护栏入口。

### 5.7 skill 入口

已建立最小 skill 文档入口，但未在初始化阶段预沉淀未经验证的 skill。

### 5.8 标准产物清单

已按 `project_init_artifacts.md` 完成最小标准产物落地。

---

## 6. 初始化阶段禁止事项

初始化阶段默认禁止以下行为：

1. 在未明确目标和非目标前先创建大量目录。
2. 为未来可能需求预埋完整平台结构。
3. 在初始化阶段直接开始复杂功能开发。
4. 把项目事实、执行规则和任务细节混写在同一文档中。
5. 因为“看起来完整”而一次性铺设过多模板和空文件。

---

## 7. 初始化完成判定

当以下条件全部满足时，可以认为项目初始化完成：

1. 当前项目目标清晰。
2. 当前 phase 和非目标清晰。
3. 最小骨架已建立。
4. `README.md`、`AGENTS.md`、`docs/facts/project_scope.md`、`docs/facts/facts_index.md` 已建立或已明确。
5. `rules/`、facts 模板层和 skill 入口已建立或已明确。
6. 后续开发流程入口已明确。

此时项目可以从初始化阶段切换到任务执行阶段。

---

## 8. 与其他 SOP 的边界

本文件只处理项目如何启动，不处理以下内容：

1. 单任务如何执行，由 `task_execution_sop.md` 负责。
2. 稳定事实如何回写，由 `fact_writeback_sop.md` 负责。
3. 项目出现偏移如何恢复，由 `maintenance_recovery_sop.md` 负责。
4. 全局原则和整体文档分工，由 `ai_project_sop.md` 负责。
5. 初始化时最小生成哪些文件和目录，由 `project_init_artifacts.md` 负责。

---

## 9. 最终结论

项目初始化的核心目标不是“把项目搭满”，而是：

> **在开始开发之前，先把目标、边界、最小骨架和治理入口建立起来。**

只有完成这一点，项目后续开发才具备可控性和可持续性。
