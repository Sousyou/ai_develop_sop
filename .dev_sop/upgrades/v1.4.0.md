# Upgrade To 1.4.0

Use this upgrade when a copied project already adopted `1.3.0` but still distinguishes project-generated rules, specs, and facts by filename prefixes instead of by directory names.

## Why This Version Exists

`1.3.0` separated baseline SOP assets from project-generated assets, but it expressed the project side partly through file naming such as `project-*.md`.

`1.4.0` keeps the same layered model and precedence contract, but moves the distinction to the directory level:

- starter-owned baseline rules remain under `.dev_sop/core/rules/*`
- project-generated rules move to `.dev_sop/project-rules/*`
- project-generated specs move to `.dev_sop/project-specs/*`
- project-generated facts move to `.dev_sop/project-facts/*`

Inside those project directories, use natural filenames.

## Intended Outcome

After this upgrade:

- `.dev_sop/project-rules/*` is the project-generated rule surface
- `.dev_sop/project-specs/*` is the project-generated spec surface
- `.dev_sop/project-facts/*` is the project-generated fact surface
- project files no longer need `project-` filename prefixes
- spec filenames under `.dev_sop/project-specs/*` use `YYYYMMDD-NNN-task-slug.md`
- baseline-vs-project distinction is expressed by directory placement, not by mixed naming rules

## Migration Map

Move surfaces as follows:

- `.dev_sop/rules/README.md -> .dev_sop/project-rules/README.md`
- `.dev_sop/rules/project-rule-index.md -> .dev_sop/project-rules/rule-index.md`
- `.dev_sop/rules/project-engineering.md -> .dev_sop/project-rules/engineering.md`
- `.dev_sop/rules/project-validation.md -> .dev_sop/project-rules/validation.md`
- `.dev_sop/rules/project-exceptions.md -> .dev_sop/project-rules/exceptions.md`
- `.dev_sop/specs/README.md -> .dev_sop/project-specs/README.md`
- `.dev_sop/facts/README.md -> .dev_sop/project-facts/README.md`
- `.dev_sop/facts/project-scope.md -> .dev_sop/project-facts/scope.md`

If copied-project repositories created additional project outputs with the old filename-prefix convention, migrate them by surface:

- `.dev_sop/rules/project-*.md -> .dev_sop/project-rules/*.md`
- `.dev_sop/specs/project-YYYYMMDD-NNN-task-slug.md -> .dev_sop/project-specs/YYYYMMDD-NNN-task-slug.md`
- `.dev_sop/facts/project-*.md -> .dev_sop/project-facts/*.md`

Do not move starter-owned baseline rules, templates, examples, or reusable skills out of `.dev_sop/core/*`.

## Required Changes

1. Create `.dev_sop/project-rules/`, `.dev_sop/project-specs/`, and `.dev_sop/project-facts/`.
2. Move the existing project-generated rules, specs, and facts into those directories.
3. Rename project-generated files to natural filenames inside those directories.
4. Remove now-empty legacy directories such as `.dev_sop/rules/`, `.dev_sop/specs/`, and `.dev_sop/facts/` when their contents have been migrated.
5. Rewrite entry docs, control docs, templates, guides, and examples so they reference the new project directories instead of the old file-prefix convention.
6. Preserve `.dev_sop/core/rules/*` as the baseline rule layer.
7. Keep the precedence contract unchanged:
   `approved exceptions > active task spec > project rules > core rules > entry adapter notes`
8. Update `.dev_sop/VERSION.md` only after the migration is complete and validated.

## Mainline Files To Update

At minimum, align all live references in:

- `README.md`
- `AGENTS.md`
- `CLAUDE.md`
- `.dev_sop/README.md`
- `.dev_sop/VERSION.md`
- `.dev_sop/control/*`
- `.dev_sop/project-rules/*`
- `.dev_sop/project-specs/*`
- `.dev_sop/project-facts/*`
- `.dev_sop/core/templates/*`
- `.dev_sop/core/guides/*`
- `.dev_sop/core/examples/*`

Historical references may remain in older upgrade notes when they are needed to explain prior migrations.

## Validation

Verify all of:

- `.dev_sop/project-rules/`, `.dev_sop/project-specs/`, and `.dev_sop/project-facts/` exist
- `.dev_sop/rules/`, `.dev_sop/specs/`, and `.dev_sop/facts/` are no longer used as live project surfaces
- files under `.dev_sop/project-rules/` use natural filenames such as `rule-index.md`, `engineering.md`, `validation.md`, and `exceptions.md`
- files under `.dev_sop/project-specs/` other than `README.md` use `YYYYMMDD-NNN-task-slug.md`
- files under `.dev_sop/project-facts/` use natural descriptive filenames
- live entry docs, templates, guides, and examples no longer tell readers to use `project-*` filename prefixes
- `.dev_sop/core/rules/*` still exists as the baseline rule surface
- `.dev_sop/VERSION.md` reports `1.4.0`

## Do Not Carry Forward

Do not preserve:

- mixed rules where the directory says one thing but filenames still imply another contract
- new live references to `.dev_sop/rules/*`, `.dev_sop/specs/*`, or `.dev_sop/facts/*` as the current project surfaces
- new instructions that require `project-*.md` naming inside the project-generated directories

## Completion Check

The upgrade is complete when:

1. baseline assets remain clearly under `.dev_sop/core/*`
2. project-generated rules, specs, and facts are distinguished by directory names
3. the current mainline docs consistently point to `.dev_sop/project-rules/*`, `.dev_sop/project-specs/*`, and `.dev_sop/project-facts/*`
4. older path conventions survive only in historical upgrade notes where they are needed for migration
