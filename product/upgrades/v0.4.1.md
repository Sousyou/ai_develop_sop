# Upgrade To 0.4.1

## Target Version
`0.4.1`

## Purpose

Use this patch upgrade when a copied project is already on the `.dev_sop/` layout but still needs the post-`0.4.0` rule sync and upgrade-guidance fixes.
This note is written so an AI agent can execute the patch directly inside a real project repository.

## Applies To

- copied projects on `0.4.0`
- copied projects that already use `.dev_sop/` but do not yet include the latest Cursor rule sync and upgrade guidance clarifications

## Summary

- sync the Cursor boundaries rule with the canonical `.dev_sop/control`, `.dev_sop/doc`, and `.dev_sop/skill` roots
- tighten the rule for when spec creation may be skipped by defining `trivially narrow` more explicitly
- clarify that the source path embedded in the SOP upgrade prompt template is a repository-local default, not a cross-platform contract
- remind Codex users to start a fresh thread after upgrades that change `AGENTS.md`, adapters, or other AI entrypoints

## End-State Contract

After the upgrade:

- `.cursor/rules/boundaries.mdc` names `.dev_sop/control`, `.dev_sop/doc`, and `.dev_sop/skill` as the canonical singular roots
- `.cursor/rules/workflow.mdc` and `.dev_sop/doc/specs/README.md` both define `trivially narrow` with explicit criteria
- `.dev_sop/doc/templates/sop-upgrade-agent-prompt-template.md` states that `F:\ai_develop_sop` is only a repository-local default source path
- upgrade guidance tells Codex users to start a fresh thread if the upgrade changed AI entrypoints
- `.dev_sop/VERSION.md` exists and states `0.4.1`

## Preflight

Before editing:

1. Read the project's `.dev_sop/VERSION.md`.
2. Inspect `.cursor/rules/boundaries.mdc` and `.cursor/rules/workflow.mdc` if they exist.
3. Inspect `.dev_sop/doc/specs/README.md` and `.dev_sop/doc/templates/sop-upgrade-agent-prompt-template.md` if they exist.
4. Identify any project-local SOP customizations in rules, specs guidance, or upgrade templates that should be preserved.

## Required Changes

1. Update `.cursor/rules/boundaries.mdc` so the canonical singular roots include `.dev_sop/control`, `.dev_sop/doc`, and `.dev_sop/skill`.
2. Update `.cursor/rules/workflow.mdc` so spec creation may be skipped only when the task is effectively spec-complete and satisfies all explicit `trivially narrow` criteria.
3. Update `.dev_sop/doc/specs/README.md` so the spec-skip guidance matches the workflow rule and names the same `trivially narrow` criteria.
4. Update `.dev_sop/doc/templates/sop-upgrade-agent-prompt-template.md` so any embedded source path such as `F:\ai_develop_sop` is clearly described as a repository-local default that may be replaced.
5. Update `AGENTS.md`, `.dev_sop/upgrades/README.md`, and any relevant upgrade prompt guidance so Codex users are told to start a fresh thread after upgrades that modify `AGENTS.md`, adapters, or other AI entrypoints.
6. Update `.dev_sop/VERSION.md` to `0.4.1` only after the patch validates.

## Preserve Rules

Do preserve:

- project-local SOP customizations that do not conflict with the target rule set
- project-owned task specs and starter-owned `example-*` reference specs
- project-local source path conventions in copied upgrade prompts when the project intentionally changed them

Do not preserve:

- outdated Cursor rule wording that still omits `.dev_sop/control` from the canonical roots
- loose spec-skip wording that no longer matches the current workflow and specs guidance
- upgrade prompt wording that treats the source path default as a fixed cross-platform requirement

## Execution Notes For AI

- Treat this as a patch upgrade on top of the existing `.dev_sop/` structure, not a namespace migration.
- When the project has already customized its upgrade prompt template, preserve the local convention while still making it clear that the path is editable and repository-local.
- If the project does not use Cursor rules, preserve that choice; do not invent `.cursor/rules/*` files solely because this starter has them.
- If the upgrade changes `AGENTS.md`, adapters, or other AI entrypoints, call out that the user should start a fresh Codex thread before relying on the updated instructions.

## Validation

- `.cursor/rules/boundaries.mdc`, if present, includes `.dev_sop/control`, `.dev_sop/doc`, and `.dev_sop/skill`
- `.cursor/rules/workflow.mdc`, if present, defines `trivially narrow` with explicit criteria
- `.dev_sop/doc/specs/README.md` defines the same `trivially narrow` criteria
- `.dev_sop/doc/templates/sop-upgrade-agent-prompt-template.md`, if present, states that the embedded source path default is repository-local and editable
- upgrade guidance tells Codex users to start a fresh thread after entrypoint-changing upgrades
- `.dev_sop/VERSION.md` exists and states `0.4.1`

## Completion Rule

The upgrade is complete only when:

1. the copied project's rule and guidance files match the end-state contract
2. preserved local customizations are still intact
3. the copied project's `.dev_sop/VERSION.md` is updated to `0.4.1`

## Merge Cautions

- do not overwrite project-local rule customizations unless they actually conflict with the target contract
- do not introduce new Cursor rule files into a project that intentionally does not use Cursor
- do not replace a project's custom source path convention unless the project wants to rebase onto the starter default
