# Upgrade To 1.6.0

Use this upgrade when a copied project already adopted `1.5.0` but the mainline surface became too minimal and no longer carries starter-owned cross-project reusable assets.

## Why This Version Exists

`1.5.0` correctly moved repository-specific project artifacts out of the starter surface, but it also moved too much cross-project reusable content into `_backup`.

`1.6.0` restores the intended split:

- keep starter-owned cross-project reusable assets on the canonical mainline under `.dev_sop/core/*`
- keep only repository-local backup and non-copyable project artifacts under `.dev_sop/_backup/*`

## Intended Outcome

After this upgrade:

- `.dev_sop/core/templates/*`, `.dev_sop/core/guides/*`, `.dev_sop/core/examples/*`, and `.dev_sop/core/skills/*` are part of the starter surface again
- `.dev_sop/_backup/*` is reserved for repository-local snapshots and project-produced artifacts that should not ship with the starter
- current mainline docs describe the fuller starter surface rather than the temporary `required-only` interpretation
- copied projects can still adopt a lean subset, but the source starter keeps the reusable assets available by default

## Migration Map

If `1.5.0` moved starter-owned reusable assets into `_backup`, restore them:

- `.dev_sop/_backup/<snapshot>/core/templates/* -> .dev_sop/core/templates/*`
- `.dev_sop/_backup/<snapshot>/core/guides/* -> .dev_sop/core/guides/*`
- `.dev_sop/_backup/<snapshot>/core/examples/* -> .dev_sop/core/examples/*`
- `.dev_sop/_backup/<snapshot>/core/skills/* -> .dev_sop/core/skills/*`

Do not restore repository-local project artifacts such as:

- `.dev_sop/_backup/<snapshot>/project-artifacts/project-rules/*`
- `.dev_sop/_backup/<snapshot>/project-artifacts/project-facts/*`
- `.dev_sop/_backup/<snapshot>/project-artifacts/control/*`

Those remain backup-only unless the current copied project explicitly wants to recover them.

## Required Changes

1. Restore starter-owned reusable assets from `_backup` to `.dev_sop/core/*` when those assets were moved off the mainline in `1.5.0`.
2. Rewrite `README.md`, `.dev_sop/README.md`, `.dev_sop/VERSION.md`, and entry adapters so they describe the starter surface as `core + project surfaces + control + upgrades`.
3. Keep `.dev_sop/_backup/*` non-canonical and repository-local.
4. Keep project-generated rule, spec, and fact surfaces empty or light by default until the copied project produces real content.
5. Update any mainline guide or example that assumes optional project-local rule files or facts already exist by default.
6. Update `.dev_sop/VERSION.md` only after the new contract validates.

## Validation

Verify all of:

- `.dev_sop/core/templates/*` contains the starter-owned templates
- `.dev_sop/core/guides/*` contains the starter-owned guides
- `.dev_sop/core/examples/*` contains the starter-owned examples
- `.dev_sop/core/skills/*` contains the starter-owned reusable skills
- `.dev_sop/_backup/*` contains only repository-local backup material or project-produced artifacts
- live mainline docs no longer describe the starter as `required-only`
- live mainline docs no longer imply `_backup` should be copied into target projects
- `.dev_sop/VERSION.md` reports `1.6.0`

## Do Not Carry Forward

Do not preserve as current-contract assumptions:

- that only the task-spec template belongs under `.dev_sop/core/templates/*`
- that starter-owned guides, examples, or skills should live only in `_backup`
- that `_backup` is part of the copyable starter package

## Completion Check

The upgrade is complete when:

1. starter-owned cross-project reusable assets are back on the canonical mainline
2. repository-local project artifacts remain outside the starter surface
3. copied projects can take the fuller starter by default or intentionally trim to a lean subset
