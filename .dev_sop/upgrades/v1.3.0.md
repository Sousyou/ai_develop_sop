# Upgrade To 1.3.0

## Target Version
`1.3.0`

## Purpose

Use this upgrade when a copied project already adopted `1.2.0` but still blurs starter-owned baseline assets with project-generated outputs.

This upgrade makes the layer contract explicit:

- starter-owned baseline rules live under `.dev_sop/core/rules/*`
- starter-owned examples live under `.dev_sop/core/examples/*`
- project-generated rule files live under `.dev_sop/rules/project-*.md`
- project-generated specs live under `.dev_sop/specs/project-*.md`
- project-generated facts live under `.dev_sop/facts/project-*.md`

## Applies To

- copied projects on `1.2.0`
- copied projects that still keep all active rules under `.dev_sop/rules/*` without a baseline layer
- copied projects that still mix project outputs and starter-owned files without a naming convention

## End-State Contract

After the upgrade:

- `.dev_sop/core/rules/*` holds the starter-owned baseline rules
- `.dev_sop/core/templates/*` holds the starter-owned blank templates
- `.dev_sop/core/examples/*` holds the starter-owned reference artifacts
- `.dev_sop/core/skills/*` holds the starter-owned reusable workflows
- `.dev_sop/rules/project-*.md` holds project-generated rule files
- `.dev_sop/specs/project-*.md` holds project-generated specs
- `.dev_sop/facts/project-*.md` holds project-generated facts
- `.dev_sop/rules/README.md`, `.dev_sop/specs/README.md`, and `.dev_sop/facts/README.md` explain the project output surfaces
- `.dev_sop/VERSION.md` exists and states `1.3.0`

## Required Moves

Apply these path migrations:

- shared baseline rules from `.dev_sop/rules/*` into `.dev_sop/core/rules/*`
- starter-owned reference facts into `.dev_sop/core/examples/facts/*`
- project-generated rules to `.dev_sop/rules/project-*.md`
- project-generated specs to `.dev_sop/specs/project-*.md`
- project-generated facts to `.dev_sop/facts/project-*.md`

## Required Changes

1. Rewrite `AGENTS.md`, `CLAUDE.md`, and `.dev_sop/README.md` so they point to baseline rules first, then project rules.
2. Add `.dev_sop/core/rules/rule-index.md` as the baseline rule entry.
3. Add `.dev_sop/rules/README.md` and `.dev_sop/rules/project-rule-index.md` as the project rule surface guides.
4. Rewrite `.dev_sop/specs/README.md` and `.dev_sop/facts/README.md` so they define `project-*` naming for real project outputs.
5. Update guides, templates, facts, and examples so they distinguish baseline assets from project-generated outputs.
6. Update `.dev_sop/VERSION.md` to `1.3.0` only after the migration validates.

## Preserve Rules

Do preserve:

- real project-generated specs, facts, and rules
- starter-owned templates, guides, examples, and reusable skills that still belong in the package
- local customizations that still make sense under the new naming and layer contract

Do not preserve:

- baseline rules mixed into the project rule surface when they can live under `.dev_sop/core/rules/*`
- project-generated specs, facts, or rules without the `project-*` naming convention
- starter-owned reference facts mixed into `.dev_sop/facts/*`

## Validation

- `.dev_sop/core/rules/*` exists and contains the baseline rule layer
- `.dev_sop/rules/README.md` exists
- `.dev_sop/rules/project-rule-index.md` exists
- files under `.dev_sop/rules/` other than `README.md` use the `project-*.md` pattern
- files under `.dev_sop/specs/` other than `README.md` use the `project-YYYYMMDD-NNN-task-slug.md` pattern
- files under `.dev_sop/facts/` other than `README.md` use the `project-*.md` pattern
- starter-owned example facts live under `.dev_sop/core/examples/facts/*`
- `.dev_sop/VERSION.md` exists and states `1.3.0`

## Completion Rule

The upgrade is complete only when:

1. baseline assets and project-generated outputs occupy the correct layers
2. project-generated outputs follow the `project-*` naming convention
3. `.dev_sop/VERSION.md` is updated to `1.3.0`
